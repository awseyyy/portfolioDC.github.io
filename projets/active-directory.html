Situation professionnelle · Atelier + Stage 2

Mise en place d'un Active Directory
et stratégies de groupe.

Déploiement complet d'un domaine techno.local sous Windows Server 2022 pour centraliser l'authentification, les droits et la configuration des postes d'une entreprise fictive de 40 utilisateurs.

Durée3 semaines
ContexteStage 2 / Atelier pro
EnvironnementWindows Server 2022
BlocBloc 1 · Bloc 2

1. Contexte & besoin

L'entreprise fictive TechnoServices compte 40 collaborateurs répartis sur trois services : Direction, Comptabilité et Production. Jusqu'ici, chaque poste était administré en workgroup : comptes locaux, mots de passe oubliés, aucune homogénéité, partages réseau désordonnés.

La direction a demandé :

  • Une authentification unique (SSO) et centralisée pour tous les postes.
  • Des droits différenciés par service (ex : la compta ne doit pas accéder aux dossiers de production).
  • Une stratégie de mot de passe conforme aux recommandations de l'ANSSI (12 caractères minimum, complexité, expiration 90 jours).
  • Des lecteurs réseau mappés automatiquement et un fond d'écran d'entreprise appliqué sur tous les postes.
  • Une résolution DNS interne pour les ressources du SI.

2. Architecture cible

┌─────────────────────────────┐ │ Internet / FAI │ └──────────────┬──────────────┘ │ WAN ┌─────────▼─────────┐ │ pfSense (GW) │ 192.168.1.1 └─────────┬─────────┘ │ LAN 10.0.0.0/24 ┌───────────────────────────┼───────────────────────────┐ │ │ │ ┌────────▼────────┐ ┌─────────▼────────┐ ┌────────▼────────┐ │ DC01 │ │ FS01 │ │ Postes clients │ │ Windows 2022 │ │ Partage SMB │ │ Win 10 / 11 │ │ AD DS + DNS │ │ \\fs01\shares │ │ joined domain │ │ 10.0.0.10 │ │ 10.0.0.20 │ │ 10.0.0.100+ │ └─────────────────┘ └──────────────────┘ └─────────────────┘
  • DC01 — contrôleur de domaine, serveur DNS intégré AD, rôle DHCP (baux 8h sur le VLAN clients).
  • FS01 — serveur de fichiers, partages SMB avec permissions NTFS héritées des groupes AD.
  • pfSense — passerelle et pare-feu (voir projet pfSense).

3. Arborescence Active Directory (OU)

L'organisation a été pensée par fonction métier, pas par hiérarchie, pour que les GPO ciblent précisément chaque besoin :

techno.local ├── OU=TECHNO │ ├── OU=Utilisateurs │ │ ├── OU=Direction → 4 comptes │ │ ├── OU=Compta → 6 comptes │ │ └── OU=Production → 30 comptes │ ├── OU=Groupes │ │ ├── GG_Direction │ │ ├── GG_Compta │ │ └── GG_Production │ └── OU=Ordinateurs │ ├── OU=Postes_Direction │ ├── OU=Postes_Compta │ └── OU=Postes_Production └── OU=Serveurs ├── DC01 └── FS01
Bonne pratique — les objets système (Users, Computers) par défaut ne peuvent pas recevoir de GPO liée. J'ai donc créé une OU racine TECHNO et déplacé tous les objets dedans dès le départ.

4. Installation pas-à-pas

4.1 — Préparation de DC01

  1. Renommage : Rename-Computer -NewName "DC01" -Restart
  2. Adressage IP statique : 10.0.0.10/24, passerelle 10.0.0.1, DNS pointant sur lui-même (127.0.0.1).
  3. Installation des mises à jour Windows.

4.2 — Promotion en contrôleur de domaine

# Ajout des rôles AD DS + DNS Install-WindowsFeature AD-Domain-Services, DNS -IncludeManagementTools # Création de la nouvelle forêt Install-ADDSForest ` -DomainName "techno.local" ` -DomainNetbiosName "TECHNO" ` -ForestMode "WinThreshold" ` -DomainMode "WinThreshold" ` -InstallDns:$true ` -DatabasePath "C:\Windows\NTDS" ` -SysvolPath "C:\Windows\SYSVOL" ` -Force

Le serveur redémarre, AD DS et DNS sont installés et fonctionnels. Je vérifie avec dcdiag et Get-ADDomain.

4.3 — Création massive des utilisateurs (PowerShell)

Plutôt que de créer 40 comptes à la main, j'ai préparé un fichier CSV users.csv puis ce script :

# Import-Users.ps1 — création en masse depuis un CSV Import-Module ActiveDirectory $users = Import-Csv -Path ".\users.csv" -Delimiter ";" foreach ($u in $users) { $sam = "$($u.Prenom.Substring(0,1).ToLower())$($u.Nom.ToLower())" $upn = "$sam@techno.local" $pwd = ConvertTo-SecureString $u.MotDePasse -AsPlainText -Force New-ADUser ` -Name "$($u.Prenom) $($u.Nom)" ` -GivenName $u.Prenom -Surname $u.Nom ` -SamAccountName $sam -UserPrincipalName $upn ` -Path "OU=$($u.Service),OU=Utilisateurs,OU=TECHNO,DC=techno,DC=local" ` -AccountPassword $pwd ` -ChangePasswordAtLogon $true ` -Enabled $true Add-ADGroupMember -Identity "GG_$($u.Service)" -Members $sam Write-Host "✔ Compte créé : $sam" -ForegroundColor Green }

Résultat : 40 utilisateurs créés en 8 secondes, tous rangés dans la bonne OU, membres du bon groupe global, mot de passe à changer à la première connexion. Gain de temps énorme et zéro erreur de saisie.

5. Stratégies de groupe (GPO)

J'ai créé et liées six GPO différentes à partir de la console gpmc.msc :

GPOCibleEffet
GPO-MotDePasseDomaine entier12 car. mini, complexité, expir. 90 j, historique 24.
GPO-FondEcranOU UtilisateursImpose wallpaper.jpg depuis \\fs01\gpo$.
GPO-MapDrivesCompta · ProductionMonte P:\ → partage service (filtré par groupe).
GPO-PanelLockOU Postes_ProductionMasque le Panneau de configuration.
GPO-USBComptaBlocage des supports USB (lecture/écriture).
GPO-Pare-FeuTous les postesActive Windows Defender Firewall & règles minimales.

Exemple de filtrage par sécurité

Pour que GPO-MapDrives ne monte le lecteur Compta que sur les utilisateurs du groupe GG_Compta, j'ai utilisé le Security Filtering natif : GPO liée à l'OU mais filtrée sur le groupe. C'est plus propre qu'un WMI Filter et ça passe mieux à l'examen.

6. Tests & recette

  • gpupdate /force puis gpresult /h report.html sur un poste de chaque service → vérif des GPO appliquées.
  • Connexion croisée (compta sur un poste prod) → le lecteur Compta suit l'utilisateur, pas le poste ✔
  • Tentative de connexion d'une clé USB sur un poste compta → refusée ✔
  • Test de mot de passe à 8 car. → rejeté ✔
Résultat — 40 utilisateurs migrés sur le domaine en une demi-journée. Plus aucun compte local administratif sur les postes clients. Les partages sont filtrés par groupes, plus par utilisateur.

8. Bilan & ce que je retiens

C'est mon projet le plus formateur : j'y ai compris la logique « OU → Groupe → GPO » et appris à scripter en PowerShell. J'ai aussi compris que le piège AD n° 1, c'est de lier des GPO sur le domaine entier au lieu de les cibler proprement. Si je devais recommencer, j'ajouterais dès le départ un second contrôleur de domaine (DC02) pour la tolérance de panne.
