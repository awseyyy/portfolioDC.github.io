Installation et durcissement
d'un serveur web LAMP sur Debian 12.

Déploiement d'un serveur web LAMP (Linux, Apache, MariaDB, PHP) sur Debian 12, avec certificat HTTPS Let's Encrypt, SSH par clé uniquement, fail2ban et UFW. Support d'un site interne WordPress.

Durée1 semaine
ContexteHome-lab
EnvironnementDebian 12 · Apache 2.4
BlocBloc 1 · Bloc 2

1. Contexte

Pour héberger l'intranet de TechnoServices, j'ai monté un serveur web from scratch. L'objectif n'était pas juste « que ça marche » mais « que ça tienne en production » : performances, sauvegarde, sécurité.

2. Installation de base

# Mise à jour système $ sudo apt update && sudo apt upgrade -y # Installation de la stack LAMP $ sudo apt install apache2 mariadb-server \ php php-mysql php-curl php-gd php-mbstring \ libapache2-mod-php unzip -y # Sécurisation MariaDB (step interactif) $ sudo mysql_secure_installation

3. Configuration d'un virtualhost

# /etc/apache2/sites-available/intranet.conf <VirtualHost *:80> ServerName intranet.techno.local DocumentRoot /var/www/intranet ErrorLog ${APACHE_LOG_DIR}/intranet-error.log CustomLog ${APACHE_LOG_DIR}/intranet-access.log combined <Directory /var/www/intranet> Options -Indexes +FollowSymLinks AllowOverride All Require all granted </Directory> </VirtualHost>
$ sudo a2ensite intranet $ sudo a2enmod rewrite headers ssl $ sudo systemctl reload apache2

4. HTTPS avec Let's Encrypt

$ sudo apt install certbot python3-certbot-apache -y $ sudo certbot --apache -d intranet.techno.fr --redirect --email admin@techno.fr --agree-tos

Certbot configure automatiquement le vhost en HTTPS, redirige tout le 80 vers le 443 et crée un cron de renouvellement. Je vérifie avec https://www.ssllabs.com/ssltest/ → note A.

5. Durcissement SSH

Par défaut, SSH accepte les mots de passe. Pour éviter le brute-force, je passe en authentification par clé uniquement :

# Sur mon poste, génération de la clé (si pas déjà fait) $ ssh-keygen -t ed25519 -C "dimitri@portfolio" # Copie sur le serveur $ ssh-copy-id dimitri@srv01 # Sur le serveur : /etc/ssh/sshd_config PasswordAuthentication no PermitRootLogin no Port 2222 AllowUsers dimitri
$ sudo systemctl restart ssh

6. Pare-feu UFW + fail2ban

$ sudo apt install ufw fail2ban -y $ sudo ufw default deny incoming $ sudo ufw default allow outgoing $ sudo ufw allow 2222/tcp # SSH (port custom) $ sudo ufw allow 80/tcp $ sudo ufw allow 443/tcp $ sudo ufw enable

fail2ban surveille /var/log/auth.log : 3 échecs de connexion SSH en 10 minutes → IP bannie 1h. Même protection pour Apache via la jail apache-auth.

7. Sauvegardes

Un script Bash en cron quotidien (02:00) :

#!/bin/bash # backup.sh — sauvegarde intranet + BDD DATE=$(date +%Y-%m-%d) DEST=/srv/backup mkdir -p $DEST # Archive du DocumentRoot tar -czf $DEST/intranet-$DATE.tar.gz /var/www/intranet # Dump BDD mysqldump --single-transaction --all-databases | gzip > $DEST/db-$DATE.sql.gz # Rotation 7 jours find $DEST -type f -mtime +7 -delete # Envoi distant (NAS via rsync/SSH) rsync -az $DEST/ backup@nas.techno.local:/backup/srv01/
Attention — un backup non testé n'est pas un backup. Une fois par mois, je restaure l'archive sur une VM de test pour vérifier que le dump SQL est bien exploitable.
