Accueil / Projets / pfSense & VPN
Situation professionnelle · Atelier pro + Home-lab

Pare-feu pfSense,
DMZ et VPN nomade.

Installation et configuration d'un pare-feu pfSense avec segmentation LAN/DMZ, règles de filtrage explicites et accès VPN distant sécurisé (OpenVPN + certificats X.509) pour le personnel en télétravail.

Durée2 semaines
ContexteAtelier pro · Home-lab
EnvironnementpfSense 2.7 CE
BlocBloc 2

1. Contexte & besoin

Dans la continuité du projet Active Directory, l'entreprise fictive TechnoServices a besoin d'un pare-feu périmétrique pour :

  • Séparer le LAN interne (postes + serveurs) du WAN (internet).
  • Héberger un serveur web public en DMZ, accessible depuis internet mais isolé du LAN.
  • Permettre aux collaborateurs en télétravail d'accéder aux ressources internes via un VPN.
  • Filtrer strictement les flux (default deny) et journaliser les connexions suspectes.

Le choix s'est porté sur pfSense CE 2.7 pour son coût nul, son interface web complète et son écosystème de paquets (openvpn, pfBlockerNG, Suricata…).

2. Architecture réseau

Internet (WAN) ─ IP publique box │ │ re0 (WAN) 192.0.2.5/24 ┌───────▼────────┐ │ pfSense │ (routeur + firewall) │ em0 em1 em2 │ └──┬──────┬──────┴─┐ │ │ │ LAN │ DMZ │ OPT │ │ │ 10.0.0.0/24│ │ 10.0.10.0/24 Postes│ │ Serveur WEB01 (Apache) AD │ │ FS01 │ │

Trois interfaces physiques (ou VLAN sur un switch) :

  • WAN (em0) — DHCP depuis la box.
  • LAN (em1) — 10.0.0.1/24, DHCP distribué aux postes.
  • DMZ (em2) — 10.0.10.1/24, statique pour les serveurs publics.

3. Règles de filtrage (philosophie)

Le principe : tout est interdit, sauf ce qui est explicitement autorisé. Sur chaque interface, la dernière règle est implicitement « deny all », donc je n'ouvre que ce qui est vraiment nécessaire.

InterfaceSrcDstPortAction
LANLAN netany80,443,53ALLOW
LANLAN netDMZ net80,443ALLOW
LANLAN netDMZ net*BLOCK
WANanyDMZ:WEB0180,443ALLOW (NAT)
DMZDMZ netLAN net*BLOCK
DMZDMZ netany80,443,53ALLOW (MAJ)
Pourquoi bloquer DMZ → LAN ? Si un jour le serveur web est compromis, l'attaquant ne doit pas pouvoir « rebondir » vers l'Active Directory. C'est le bon réflexe sécurité d'une DMZ.

4. NAT (redirection de port)

Pour que le site public soit accessible depuis internet, j'ai créé un Port Forward dans Firewall → NAT → Port Forward :

# Interface : WAN # Protocole : TCP # Destination : WAN address # Dest. port : HTTP (80), HTTPS (443) # Redirect target IP : 10.0.10.10 (WEB01) # Redirect target port : 80 / 443

pfSense crée automatiquement la règle associée sur l'interface WAN. Je vérifie avec Diagnostics → Packet Capture qu'un client internet atteint bien WEB01.

5. VPN OpenVPN (accès nomade)

5.1 — Préparation de l'autorité de certification

  1. System → Cert. Manager → CAs → création d'une CA VPN-TECHNO-CA (RSA 2048, SHA-256).
  2. Certificates → certificat serveur vpn-srv signé par la CA.
  3. Un certificat utilisateur par collaborateur nomade.

5.2 — Assistant OpenVPN

J'ai utilisé l'assistant (VPN → OpenVPN → Wizards) pour créer un serveur :

  • Mode : Remote Access (SSL/TLS + User Auth)
  • Protocole : UDP sur port 1194
  • Réseau tunnel : 10.8.0.0/24
  • Réseaux poussés au client : 10.0.0.0/24 (LAN)
  • DNS poussé : 10.0.0.10 (DC01)

5.3 — Export du client

Grâce au paquet openvpn-client-export, je génère pour chaque utilisateur un installateur .exe ou un profil .ovpn qui embarque déjà la CA, le certif client et la clé. L'utilisateur n'a plus qu'à double-cliquer et saisir son identifiant AD (authentification combinée cert + login).

6. Tests

  • Ping LAN → DMZ : refusé ✔
  • HTTP LAN → WEB01 : OK ✔
  • Accès public au site depuis un téléphone en 4G : OK ✔
  • Connexion VPN depuis un réseau externe puis accès à un partage \fs01\shares : OK ✔
  • Test de filtrage : un client VPN essaie de pinger la box internet du siège → bloqué ✔

8. Bilan

pfSense m'a vraiment fait comprendre la logique des pare-feu modernes : stateful inspection, default deny, NAT sortant automatique, règles par interface. J'ai aussi appris à lire les logs pf pour expliquer pourquoi un flux est bloqué. La prochaine étape logique serait d'installer Suricata (IDS/IPS) sur l'interface WAN pour détecter les scans.
